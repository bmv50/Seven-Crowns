## Слоистая модель

```
┌─────────────────────────────────────────────────────┐
│  TELEGRAM-СЛОЙ (bot/)                                 │
│  aiogram 3 · хендлеры · callback'и · inline-UI        │
│  ── тонкий, знает про движок, но не наоборот ──        │
└───────────────────────────┬───────────────────────────┘
                            │ вызовы
┌───────────────────────────┴───────────────────────────┐
│  ДВИЖОК (engine/)                                      │
│  чистая механика, НЕ знает про Telegram.               │
│  Единственный арбитр правил.                           │
│  content · character · world · combat · loop · quest · │
│  social · db                                           │
└───────────────────────────┬───────────────────────────┘
                            │ читает
┌───────────────────────────┴───────────────────────────┐
│  ДАННЫЕ (data/*.yaml)  +  PostgreSQL                   │
│  контент в YAML (валидируется) · персонажи в БД        │
└─────────────────────────────────────────────────────┘

         ⌐ — — — — — — — — — — — — — — — — — — ¬
         | ИИ-СЛОЙ (ai/) — БУДУЩЕЕ, подключается |
         | СБОКУ к движку, не внутрь. Возвращает  |
         | текст или JSON-предложения, которые    |
         | движок валидирует.                     |
         L — — — — — — — — — — — — — — — — — — — ⌐
```

**Почему так:** движок можно тестировать и переиспользовать (хоть с веб-интерфейсом, хоть с Discord). Telegram — деталь представления. ИИ — опциональная надстройка. Игра работает без ИИ полностью.

## Поток данных

### При старте
1. `bot/main.py` грузит `.env` (токен, БД, прокси).
2. `engine/content.py` загружает и **валидирует** все YAML (целостность ссылок: выходы комнат, спавны, лут, скиллы, квесты).
3. `engine/db.py` подключается к PostgreSQL, создаёт/мигрирует схему, грузит персонажей в память.
4. `engine/world.py` спавнит мобов из `world.yaml` в рантайм.
5. `engine/loop.py` запускает фоновый тик-цикл (1 раз/сек).
6. Бот стартует polling.

### При действии игрока (нажал кнопку)
1. Telegram → callback → `bot/main.py` (`on_cb`).
2. Хендлер вызывает функцию движка (атака, перемещение, скилл...).
3. Движок меняет состояние (HP моба, позиция, инвентарь), возвращает текст + клавиатуру.
4. Если есть другие игроки в комнате — `loop.broadcast` рассылает им событие.
5. Изменения персонажа сохраняются в БД (`db.save`).
6. Бот редактирует сообщение (новый текст + кнопки).

### В фоновом тике (раз/сек, `loop.tick`)
1. По всем комнатам: мобы с агро, у кого вышел `tick_speed`, атакуют случайного игрока из агро-листа.
2. Тикают эффекты на мобах (яд).
3. Смерть моба → награды киллерам (делёж опыта/золота/лута), прогресс квестов, левелап.
4. Смерть игрока → телепорт в храм, рассылка.
5. Респавн мёртвых мобов по таймеру.

## Модель персонажа (`character.py`)

Характеристики **вычисляются**, не хранятся напрямую (кроме HP/MP/уровня/опыта/золота):

```
итоговый_атрибут = базовый(класс) + рост(класс, уровень) + модификатор(раса) + бонусы(экипировка) + эффекты
max_hp = (база_класса + рост) + Сила + Дух/2, всё × hp_mod(раса)
max_mp = (база_класса + рост) + Интеллект + Дух, всё × mp_mod(раса)
атака   = основной_атрибут(класс) + бонус_оружия + уровень
защита  = броня + Ловкость/3
крит    = 0.05 + Ловкость×0.005 + бонус_класса + бонус_расы
вампиризм = lifesteal_класса + lifesteal_расы
```

Это значит: смена экипировки/расы/уровня автоматически пересчитывает всё. Менять формулы — только здесь.

## Схема БД (PostgreSQL)

Таблица `characters`:
- скалярные поля: `uid, name, cls, race, room, level, xp, hp, mp, gold, updated_at`
- JSONB: `equipment, inventory, quests, flags`
- автосоздание при старте; миграция через `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`.

**Что НЕ в БД (живёт в памяти):** HP мобов, таймеры респавна, агро, активные эффекты, состояние пати/дуэлей. Возрождается из YAML при рестарте. Это осознанный выбор: рантайм-состояние мира эфемерно, прогресс игроков персистентен.

**Будущее (ИИ-память):** расширение `pgvector` + таблица `memories` (npc_id, player_uid, content, embedding, importance) для RAG-памяти NPC — в той же БД, без новой инфраструктуры.

## Конфигурация (`.env`)

- `BOT_TOKEN` — от BotFather.
- `DATABASE_URL` — строка подключения PostgreSQL.
- `PROXY_URL` — HTTP-прокси для обхода блокировки Telegram (опционально).
- `AI_PROVIDER` + параметры — ИИ-слой диалогов NPC (`none` | `deepseek`; см. `ai/provider.py`).

Заложена **абстракция провайдера ИИ**: переключение провайдера одной переменной, без правки кода. Реализован DeepSeek (OpenAI-совместимый API); при выключенном ИИ NPC отвечают шаблонными `lines` — игра полноценна без ИИ.

## Принципы, которых держимся

1. **Движок — арбитр.** ИИ никогда не меняет состояние напрямую; только предлагает, движок валидирует.
2. **Деградация без ИИ.** Игра полноценна без ИИ-слоя.
3. **ИИ вне горячего пути.** Боевой тик детерминирован; ИИ-вызовы асинхронны и редки.
4. **Ввод игрока — недоверенные данные.** Изоляция от системных промптов (защита от инъекций) — на будущее.
5. **Контент в данных.** Мир расширяется правкой YAML + валидатор, а не кодом.
6. **Необратимые решения — на старте правильно.** Абстракция провайдера, схема памяти, embedding-модель, логирование — закладываются заранее, чтобы не переделывать.
