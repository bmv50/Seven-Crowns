## ИИ-СЛОЙ (Этап 0 → дальше, по дорожной карте из анализа)

Целевая модель — **Модель A (контролируемая)**: ИИ ведёт диалоги NPC + событийный «бог», движок остаётся арбитром.

### Этап 0-ИИ. Фундамент абстракций (необратимые решения — сделать правильно)
1. **Пакет `ai/`** с абстракцией провайдера: интерфейс `generate(prompt, tier)` поверх всех моделей. Tier: cheap/mid/premium. Реализации: `none` (заглушка→шаблоны), `ollama` (локально), `openai_compatible` (облако/OpenRouter).
2. **Установить Ollama** на ноут (RTX 4060 8GB) — модель 7-8B в Q4 (Llama 3.1 8B / Qwen). Бесплатная локальная разработка ИИ-NPC.
3. **Структурное логирование** каждого LLM-вызова (токены, стоимость, латентность, исход) — с первого дня. Langfuse self-host или таблица в Postgres.
4. **Интерфейс `MemoryStore`** поверх pgvector + **зафиксировать embedding-модель** (смена потом дорогая — переэмбеддинг всей памяти). Scoping: npc_id / player_id / session / world.

### Этап 1-ИИ. NPC-диалоги
- При «поговорить» → собрать промпт (персона NPC из YAML + RAG-память + состояние) → дешёвая/локальная модель → текст реплики.
- Откат на шаблонные `NPC_LINES` при сбое.
- Простой роутер по типу: рядовой моб → state-машина (бесплатно); именной NPC → mid-модель.
- Кэширование промптов (фиксированная персона = кэшируемый префикс, −90% на входе).

### Этап 2-ИИ. Память NPC (pgvector + RAG)
- Запись значимых событий (разговор, квест, поступок) → эмбеддинг → `memories`.
- Извлечение top-k релевантных воспоминаний в промпт (RAG вместо длинного контекста).
- Забывание: понижение важности давних, архивация.
- Трёхуровневая модель: краткосрочная (буфер) / эпизодическая (события NPC↔игрок) / фактическая (лор, RAG по YAML).

### Этап 3-ИИ. Бог-оркестратор (события)
- Таймер (раз в 5–15 мин, не на каждое сообщение) → сводка мира → 0–1 событие в JSON.
- **Валидатор**: событие из белого списка, зона существует, параметры в диапазоне → применить + разослать анонс.
- Каталог событий: нашествие, засуха (дебафф зоны), праздник (бафф/скидки), сгущение тумана (маркер прогресса).

### Этап 4-ИИ. Локализация
- i18n системного текста (ключи по локали игрока, поле `lang`). Мгновенно, бесплатно.
- Перевод живого чата на лету — **только при реальном многоязычном спросе** (дорого, с задержкой). Кэш переводов.

## МАСШТАБ И ПРОДАКШЕН (по порогам числа игроков)

- **Деплой на VPS:** Docker-compose (игра + Postgres). Бот через systemd/PM2. На проде прокси не нужен (нормальный доступ к Telegram).
- **Порог self-hosted SLM (~10k игроков):** когда per-token-счёт за диалоги > аренды GPU (~$130–400/мес RTX 4090) — поднять локальную модель для массовых NPC, премиум оставить «богу» и сюжетным NPC.
- **Порог managed-памяти:** только если pgvector упрётся в латентность на многих миллионах векторов.
- **Наблюдаемость:** SLO (латентность диалога <3с, доля отклонённых JSON <5%), дашборд, бюджетные алерты.

## ДАЛЬНИЙ ГОРИЗОНТ (опционально, с оговорками)

### Полная автономия ИИ (Модель B)
Агенты с инициативой, целями, памятью, влиянием на мир. **Моя рекомендация: не как стартовая модель.** Риски: расходы без потолка (агенты «думают» без игроков), потеря баланса/предсказуемости, тяжёлая отладка эмерджентности, расширенная поверхность атак. Вводить **точечно, за фиче-флагом**, накопив телеметрию на Модели A. Обязательные предохранители описаны в `SPEC_AI_World.md`.

### Блокчейн-предметы (NFT)
Обсуждалось — **не рекомендую на текущем этапе.** Технически возможно (custodial-мост на L2, off-chain игра + минт по запросу), но: блокчейн ломает откат/баланс (нельзя понерфить заминченный предмет), юридическая зона, безопасность ключей = финансовая ответственность, pay-to-win. Делать **только при реальном спросе на владение**, после рабочей игры, с аудитом контракта и юристом.

## Чего НЕ делать (антипаттерны, выявленные по ходу)

- **Не тащить чужие «универсальные ядра»** (типа ECC и подобных) — они про другое, добавляют вес и риск, не закрывают наши задачи.
- **Не давать ИИ прямой доступ к БД/состоянию** — только предложения через валидатор.
- **Не ставить ИИ в боевой тик** — латентность убьёт реалтайм.
- **Не строить enterprise-оркестрацию/маршрутизаторы раньше времени** — для инди-масштаба достаточно своего кода и правил.
- **Не переэмбеддивать память из-за смены модели** — зафиксировать embedding-модель на старте.

## Резюме видения

Сейчас у нас крепкий детерминированный движок. Правильный путь: **доделать пати/дуэли → отполировать → заложить абстракции ИИ (провайдер, память, логи) → диалоги NPC локально на Ollama → память на pgvector → событийный бог → локализация → продакшен на VPS**. ИИ наслаивается сбоку, движок остаётся арбитром. Полная автономия и блокчейн — далёкий опциональный горизонт, только при доказанной необходимости.
