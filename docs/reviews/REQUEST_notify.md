# Peer-ревью: архитектура push-реактивации для Telegram-MUD «Эхо Глубин»

## Контекст
Telegram-MUD на Python (aiogram 3, asyncpg, один процесс). Слоистая архитектура:
`engine/` не знает про Telegram, `bot/` — тонкий слой. Игровой цикл — asyncio-таск,
тикает 1 раз/сек (engine/loop.py). Персонажи персистятся в Postgres после каждого
значимого действия; онлайн-игроки живут в памяти (dict chars). Сейчас бот НИКОГДА
не пишет первым — все анонсы (мировые боссы и т.п.) уходят только игрокам из памяти
процесса. Retention-механики уже есть: ежедневки (ротация по дате), сезоны-лиги
(неделя), кулдауны данжей (30–60 мин), «отдохнувший опыт» с капом, аукцион.

## Предлагаемый дизайн

**engine/notify.py** (чистый, без aiogram):
- Каталог категорий: daily_reset, world_boss, auction_sold, auction_outbid,
  dungeon_ready, rested_full, season_end_soon, season_rollover, guild_event.
- Настройки игрока в ch.flags["notify"] (вкл/выкл по категориям, дефолт — всё вкл).
- Анти-спам: ≤2 push/сутки на игрока (кроме критичных: auction_sold),
  тихие часы 23:00–09:00 по серверу (v1; per-player TZ — later).
- Очередь отложенных: если тихие часы/лимит — отложить до утра или дропнуть
  (по категории: world_boss протухает, daily_reset — переносится).

**БД:** колонка last_seen + notify_blocked в characters; таблица notify_schedule
(uid, category, fire_at) для персональных таймеров оффлайн-игроков
(dungeon_ready, rested_full, daily_reset 09:00). Опрос раз в 60 сек одним SELECT.

**bot/main.py:** broadcast_all(text, category) — итерация по uid из БД (не по памяти),
батчи с rate-limit ≤25 msg/sec, 403 от Telegram → notify_blocked=true (больше не слать).

**Хуки в loop.py:** спавн мирового босса → broadcast_all; ролловер сезона → то же.

**Флаг:** NOTIFY=1 (env), по умолчанию выключено, деградация без БД — тихо ничего.
