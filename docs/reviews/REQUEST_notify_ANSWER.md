Итак, по порядку:

### 1. Риски и дыры в дизайне
*   **Массовые уведомления (world_boss):** Рассылка 1–5 тыс. сообщений при 25 msg/sec займет 2–3.3 минуты. Для тайм-сенситивного `world_boss` это неприемлемо, игроки увидят его в игре до получения пуша.
*   **Анти-спам (`≤2 push/сутки`):** Слишком жестко. Игрок может пропустить важные уведомления (например, `guild_event`) из-за менее важных (например, `rested_full`), если они выбрали эти категории. Что, если ему пришло два `auction_outbid` по разным лотам? Это "съест" лимит.
*   **Тихие часы (`23:00–09:00` по серверу):** Критический UX-флоу. Для глобальной игры это означает, что игроки в других часовых поясах будут получать уведомления глубокой ночью.
*   **Опрос `notify_schedule`:** Один `SELECT` по всем `fire_at` раз в 60 секунд для тысяч записей может стать тяжелым и медленным запросом при росте числа игроков/уведомлений.
*   **Неконсистентность онлайн/оффлайн:** Не описан механизм миграции таймеров из памяти (для онлайн-игроков) в `notify_schedule` (для оффлайн) при выходе игрока или рестарте. Потенциальная потеря уведомлений.

### 2. Что бы ты сделал иначе и почему
*   **Массовые уведомления:** Для критически тайм-сенситивных событий (типа `world_boss`) не рассылать push *всем*. Отправлять только тем, кто *не был* онлайн за последние 5-15 минут. Онлайн игроки получат объявление в игре. Это значительно снизит нагрузку и время рассылки.
*   **Анти-спам:**
    *   Ввести **персональный лимит** на пуши, настраиваемый игроком. Жесткий лимит `≤2 push/сутки` — только как дефолт, который можно изменить в настройках.
    *   Разделять лимиты по *приоритету* или *типу* уведомлений (например, "игровые" vs "экономические/социальные"), а не просто категории.
    *   **Приоритетно реализовать `per-player TZ`**. Пока его нет, лучше отключить тихие часы, чтобы не вредить UX.
*   **`notify_schedule`:** Вместо одного большого `SELECT` раз в 60 секунд, использовать более эффективный подход: например, триггеры при выходе игрока для сохранения таймеров, а воркер пусть постоянно запрашивает *только ближайшие N* уведомлений к отправке. Можно также рассмотреть Redis Streams или отдельные очереди для каждого типа персональных уведомлений.
*   **Деградация при рестарте:** Явно проработать логику: при выходе онлайн-игрока все его активные таймеры (dungeon_ready, rested_full) должны быть записаны в `notify_schedule`. При незапланированном рестарте теряются только таймеры для игроков, бывших онлайн *в момент рестарта* и чьи таймеры не успели сброситься в БД.

### 3. Чего не хватает
*   **Per-player TZ:** Абсолютно критично для тихих часов.
*   **Разграничение уведомлений для онлайн/оффлайн:** Четкая политика, кто получает пуш, если уже в игре.
*   **Гибкое управление приоритетом:** Возможность для игрока отключать/включать категории, а также устанавливать для них пороги.
*   **История уведомлений:** Журнал отправленных пушей (uid, category, timestamp, sent_status) для отладки и аудита.
*   **Аналитика:** Метрики по открываемости и возврату игроков после пушей.

### 4. Что хорошо
Слоистая архитектура, разделение `engine`/`bot`, `NOTIFY` флаг, батчинг с rate-limit, базовая обработка 403.
