Business Mirror: общая архитектура, модули и технологический стек
Статус: сборочный архитектурно-технический документ.
Дата сборки: 2026-07-07.
Основание: текущий контекст Business Mirror, включая универсальную архитектуру, карту стыков, дерево пересечений, модель активной оркестрации, адаптивные редакторы, производственный MVP, модель ТОиР/ТАИР и предфинальную объектно-узловую модель Основного создания ценности (B06).
Важно: в текущих материалах утверждена доменная, логическая и интеграционная архитектура. Конкретные фреймворки, СУБД, брокеры сообщений, облачный провайдер и финальный API-транспорт пока не утверждены. Поэтому технологический стек ниже разделен на:

утвержденный логический стек;
требования к физическому стеку;
проектные кандидаты для MVP, требующие отдельного утверждения.

1. Назначение системы
Business Mirror - платформа цифрового делового зеркала бизнеса.
Система должна построить связанную электронную модель конкретного бизнеса:

людей;
ролей;
зон ответственности;
процессов;
технологических операций;
объектов создания ценности;
ресурсов;
документов;
договоров;
денег;
обязательств;
рисков;
фактов исполнения;
доверенных показателей;
внешних и внутренних стыков.

Business Mirror не является отдельной CRM, ERP, EAM, MES, WMS, ЭДО, бухгалтерией, банком или маркетплейсом. Эти системы могут быть источниками или внешними контурами. Центр Business Mirror - единое информационное поле создания, сохранения, восстановления и подтверждения добавленной стоимости.
Базовая цепочка системы:
textмодель бизнеса
-> план и сценарии
-> исполнимый контур
-> ролевые АРМ
-> прямые факты
-> подтвержденное состояние
-> деньги, риск, ресурс, качество, документы
-> обратный пересчет модели
Главная архитектурная идея:
textодна связанная модель бизнеса
-> много ролевых и функциональных проекций
-> единое происхождение факта
-> селективный обмен между слоями
-> управляемая глубина расчета
-> человек или орган управления принимает значимые решения

5.5. API-архитектура
Финальный транспорт API не утвержден. Логически нужны четыре класса API.
Командные API
Меняют состояние:

создать модель;
добавить процесс;
назначить роль;
зафиксировать факт;
изменить статус объекта;
создать сценарную копию;
утвердить изменение;
закрыть ремонтную операцию.

Правило: команда должна проверять полномочия, версию модели, применимость действия и конфликт статусов.
Query API
Читают ролевую проекцию:

получить АРМ пользователя;
получить статус объекта;
получить задачи смены;
получить карту процесса;
получить историю дефекта;
получить финансовый след действия;
получить доверенный показатель.

Правило: ответ должен быть фильтрован по роли, зоне ответственности, правам, уровню риска и режиму раскрытия.
Event API
Фиксируют или отдают события:

факт операции;
измерение;
отклонение;
подтверждение;
ручная корректировка;
изменение модели;
решение;
репликация;
конфликт версий.

Правило: значимое событие не затирает предыдущее состояние, а создает новую запись.
Integration API
Связывает Business Mirror с внешними системами:

1С;
ERP;
CRM;
MES;
EAM;
SCADA;
WMS;
ЭДО;
файловые источники;
приборы и датчики.

Правило: каждый импорт должен иметь источник, время, версию, уровень доверия, преобразование и ограничение применимости.
6. Безопасность, права и доверие
6.1. Модель доступа
Business Mirror требует комбинации RBAC и ABAC.
RBAC задает:

роль;
должность;
группу;
уровень управления;
базовые разрешения.

ABAC задает контекст:

участок;
объект;
процесс;
операция;
документ;
договор;
тип риска;
класс данных;
режим раскрытия;
текущий статус;
доверие к данным.

Пример:
textпользователь может видеть не весь ремонтный контур,
а только работы своего участка,
по объектам своей зоны ответственности,
в пределах своей роли,
без закрытой финансовой или договорной части,
если она не нужна для действия.
6.2. Защита фактов
Для значимых фактов нужны:

добавляемые журналы событий;
контрольные суммы;
цепочки контрольных сумм;
цифровые подписи значимых источников;
доверенная временная метка;
неизменяемый архив;
изолированные резервные копии;
разделение административных полномочий;
аудит чтения, изменения и восстановления.

Администратор инфраструктуры не должен иметь возможность незаметно изменить подтвержденную бизнес-историю.
6.3. Режимы раскрытия данных
Данные делятся на:

внутренние данные заказчика;
ролевые данные пользователя;
данные подрядчика в гостевом АРМ;
данные для контрагента по договору;
обезличенные данные платформы;
доверенный внешний профиль;
регламентированную или юридически значимую проекцию.

Каждое раскрытие должно иметь:

получателя;
основание;
объем;
срок;
уровень доверия;
ограничения использования;
журнал раскрытия;
возможность отзыва, если режим это допускает.

7. Режимы работы и отказоустойчивость
7.1. Онлайн-режим
В онлайн-режиме:

АРМ получает актуальный контекст;
факты сразу уходят в локальный контур;
локальный контур синхронизируется с контуром предприятия;
разрешенные обезличенные данные проходят через шлюз;
платформенный контур возвращает справочники, методики и обновления.

7.2. Офлайн-режим
При потере связи локальный контур продолжает разрешенную работу.
Он должен:

фиксировать факты;
сохранять порядок событий;
применять локальные утвержденные ограничения;
хранить очередь неподтвержденных записей;
предотвращать повторную отправку;
отмечать отсутствие центрального подтверждения;
запрещать действия, требующие актуального внешнего решения.

После восстановления:
textлокальный журнал
-> проверка целостности
-> проверка версий модели
-> выявление конфликтов
-> синхронизация непротиворечивых фактов
-> разрешение конфликтов человеком
7.3. RPO и RTO
Для каждого класса данных задаются:

допустимый объем потери последних данных;
допустимое время восстановления;
минимальное количество независимых копий;
максимальный автономный период;
требуемая точка восстановления;
допустимый режим деградации.

Классы данных:

факты опасных операций;
управленческие решения;
финансовые факты;
текущие задания;
история оборудования;
справочные модели;
документы и медиа;
аналитические производные.

8. Границы MVP
8.1. Входит в MVP
MVP должен закрыть базовое цифровое зеркало и рабочую исполнимость.
Входит:

создание модели бизнеса через диалог и импорт;
редактор общей модели;
редактор ролей и зон ответственности;
редактор процессов;
редактор технологических схем;
базовая библиотека Lego-элементов;
локальные экземпляры объектов;
ролевые АРМ;
фиксация прямых фактов;
статусы объектов, операций, дефектов и ремонтов;
доказательная история;
базовая сценарная копия модели;
базовый расчет производственной или сервисной программы;
базовый контур ТОиР/ТАИР;
интеграционный шлюз для ограниченного набора источников;
базовая модель доверия и качества данных.

8.2. Не входит в MVP
Не входит:

полное покрытие всех отраслей;
полная физико-химическая цифровая модель каждого процесса;
автоматическое принятие управленческих решений;
полная банковская, страховая, налоговая и торговая площадка;
замена ERP, MES, WMS, SCADA, EAM и ЭДО;
полноценная публичная аналитика по всем отраслям;
финальная многоуровневая графовая оптимизация всех бизнес-контуров;
полная автоматическая юридическая квалификация договоров и претензий.
